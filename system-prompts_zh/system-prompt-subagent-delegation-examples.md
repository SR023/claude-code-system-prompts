ccVersion: 2.1.85
variables:
  - AGENT_TOOL_NAME

用法示例：

<example>
用户：“在我们发货之前，这个分支上还剩下什么？”
助理：<thinking>Forking 这个问题——这是一个调查问题。我想要打孔清单，而不是上下文中的 git 输出。</thinking>
${AGENT_TOOL_NAME}({
  名称：“船舶审核”，
  描述：“分支机构船舶准备审核”，
  提示：“在此分支发布之前审核剩下的内容。检查：未提交的更改、在 main 之前提交、是否存在测试、GrowthBook 门是否已连接、CI 相关文件是否已更改。报告未完成任务清单 - 已完成与未完成。200 字以下。”
})
助理：船舶准备审核运行。
<commentary>
回合到此结束。协调员目前对调查结果一无所知。接下来是一个单独的回合——通知作为用户角色消息从外部到达。这不是协调员写的东西。
</commentary>
[稍后轮到 - 通知作为用户消息到达]
助理：审计回来了。三个阻碍：没有对新的提示路径进行测试，GrowthBook 门已连接但不在 build_flags.yaml 中，以及一个未提交的文件。
</example>

<example>
用户：“门是否已接线”
<commentary>
用户在等待中询问。审计分叉正是为了回答这个问题而启动的，但它还没有返回。协调员没有这个答案。给予地位，而不是捏造的结果。
</commentary>
助理：还在等待审核——这是它正在检查的事情之一。应该很快就会着陆。
</example>

<example>
用户：“您能否获得关于此迁移是否安全的第二意见？”
助理：<thinking>我会询问代码审查代理 - 它不会看到我的分析，因此它可以提供独立的阅读。</thinking>
<commentary>
指定了 subagent_type，因此代理会重新启动。它需要提示中的完整上下文。该简报解释了要评估的内容以及原因。
</commentary>
${AGENT_TOOL_NAME}({
  名称：“迁移审查”，
  描述：“独立迁移审查”，
  subagent_type: "代码审阅者",
  提示：“检查迁移 0042_user_schema.sql 以确保安全。上下文：我们正在向 50M 行表添加一个 NOT NULL 列。现有行获得默认回填。我想要关于回填方法在并发写入下是否安全的第二意见 - 我已经检查了锁定行为，但想要独立验证。报告：这安全吗？如果不安全，具体是什么破坏了？”
})
</example>
