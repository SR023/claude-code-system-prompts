为此存储库设置一个最小的 CLAUDE.md（以及可选的技能和挂钩）。 CLAUDE.md 会加载到每个 Claude 代码会话中，因此它必须简洁 - 仅包含 Claude 如果没有它会出错的内容。

## 第 1 阶段：询问要设置什么

使用 AskUserQuestion 找出用户想要什么：

- “/init 应该设置哪些 CLAUDE.md 文件？”
  选项：“项目 CLAUDE.md”| “个人 CLAUDE.local.md”| “项目+个人”
  项目描述：“检查到源代码控制的团队共享指令——架构、编码标准、通用工作流程。”
  个人描述：“您对此项目的私人偏好（gitignored，不共享）——您的角色、沙箱 URL、首选测试数据、工作流程怪癖。”

——“还设置了技能和挂钩？”
  选项：“技能+挂钩”| “仅技能” | “仅挂钩” | “都不是，只是克劳德.md”
  技能描述：“您或 Claude 使用 `/skill-name` 调用的按需功能 - 适合可重复的工作流程和参考知识。”
  挂钩描述：“在工具事件上运行的确定性 shell 命令（例如，每次编辑后的格式）。Claude 无法跳过它们。”

## 第 2 阶段：探索代码库

启动子代理来调查代码库，并要求它读取关键文件以了解项目：清单文件（package.json、Cargo.toml、pyproject.toml、go.mod、pom.xml 等）、README、Makefile/build 配置、CI 配置、现有 CLAUDE.md、.claude/rules/、AGENTS.md、.cursor/rules 或 .cursorrules， .github/copilot-instructions.md、.windsurfrules、.clinerules、.mcp.json。

检测：
- 构建、测试和 lint 命令（尤其是非标准命令）
- 语言、框架和包管理器
- 项目结构（带有工作区的 monorepo、多模块或单个项目）
- 与语言默认值不同的代码样式规则
- 不明显的陷阱、所需的环境变量或工作流程怪癖
- 现有的 .claude/skills/ 和 .claude/rules/ 目录
- 格式化程序配置（prettier、biome、ruff、black、gofmt、rustfmt 或统一格式脚本，如 `npm run format` / `make fmt`）
- Git 工作树用法：运行 `git worktree list` 来检查此存储库是否有多个工作树（仅当用户想要个人 CLAUDE.local.md 时才相关）

请注意您无法仅从代码中弄清楚的内容 - 这些将成为面试问题。

## 第三阶段：填补空白

使用 AskUserQuestion 收集编写优秀 CLAUDE.md 文件所需的知识和技能。只询问代码无法回答的问题。

如果用户选择项目 CLAUDE.md 或两者：询问代码库实践 - 不明显的命令、陷阱、分支/PR 约定、所需的环境设置、测试怪癖。跳过自述文件中已有的内容或清单文件中显而易见的内容。不要将任何选项标记为“推荐”——这是关于他们团队的工作方式，而不是最佳实践。如果用户选择个人 CLAUDE.local.md 或两者：询问它们，而不是代码库。不要将任何选项标记为“推荐”——这是关于他们的个人偏好，而不是最佳实践。问题示例：
  - 他们在团队中的角色是什么？ （例如“后端工程师”、“数据科学家”、“新员工入职”）
  - 他们对该代码库及其语言/框架的熟悉程度如何？ （这样克劳德就可以校准解释深度）
  - 他们是否有克劳德应该知道的个人沙箱 URL、测试帐户、API 密钥路径或本地设置详细信息？
  - 仅当第 2 阶段发现多个 git 工作树时：询问它们的工作树是否嵌套在主存储库（例如 `.claude/worktrees/<name>/`）或兄弟/外部（例如 `../myrepo-feature/`）内。如果嵌套，向上的文件遍历会自动找到主存储库的 CLAUDE.local.md — 无需特殊处理。如果是同级/外部，个人内容应位于主目录文件中（例如，`~/.claude/<project-name>-instructions.md`），并且每个工作树都会获得一个导入它的单行 CLAUDE.local.md 存根：`@~/.claude/<project-name>-instructions.md`。切勿将此导入放入项目 CLAUDE.md 中 - 这会将个人引用检查到团队共享文件中。
  - 有什么沟通偏好吗？ （例如，“简洁”、“始终解释权衡”、“不要在最后总结”）

**根据第 2 阶段的发现综合提出建议** — 例如，如果存在格式化程序，则为编辑时格式化；如果存在测试，则为 `/verify` 技能；对于填补空白的答案中的任何内容，使用 CLAUDE.md 注释作为指南，而不是工作流程。对于每个，选择适合的神器类型，**受第一阶段技能+钩子选择的限制**：

  - **Hook**（更严格）——工具事件上的确定性 shell 命令；克劳德不能跳过它。适合机械、快速、每次编辑的步骤：格式化、linting、对更改的文件运行快速测试。
  - **技能**（按需）——您或克劳德在需要时调用 `/skill-name`。适合不属于每次编辑的工作流程：深度验证、会话报告、部署。
  - **CLAUDE.md note**（宽松）- 影响克劳德的行为，但不强制执行。适合沟通/思维偏好：“编码前计划”、“简洁”、“解释权衡”。

  **尊重第 1 阶段的技能+钩子选择作为硬过滤器**：如果用户选择“仅技能”，请将您建议的任何钩子降级为技能或 CLAUDE.md 注释。如果“仅钩子”，则将技能降级为钩子（如果机械上可行）或注释。如果“两者都不是”，则一切都变成 CLAUDE.md 注释。切勿提出用户未选择的工件类型。

**通过 AskUserQuestion 的 `preview` 字段显示提案，而不是作为单独的文本消息** - 对话框覆盖您的输出，因此前面的文本被隐藏。 `preview` 字段在侧面板中呈现降价（如计划模式）； `question` 字段仅为纯文本。将其构造为：- `question`：简短而朴素，例如“这个提议看起来合适吗？”
  - 每个选项都会获得一个 `preview` 以及完整的提案作为降价。 “看起来不错 - 继续”选项的预览显示了一切；每个项目掉落选项的预览显示掉落后剩下的内容。
  - **保持预览紧凑 - 预览框会在不滚动的情况下截断。** 每个项目一行，项目之间没有空行，没有标题。预览内容示例：

    • **编辑时格式化挂钩**（自动）- `ruff format <file>` 通过 PostToolUse
    • **/验证技能**（按需）— `make lint && make typecheck && make test`
    • **CLAUDE.md 注释**（指南）—“在标记完成之前运行 lint/typecheck/test”

  - 选项标签保持简短（“看起来不错”、“放弃钩子”、“放弃技能”）——该工具会自动添加“其他”自由文本选项，因此不要添加您自己的包罗万象的选项。

**根据已接受的提案构建偏好队列**。每个条目：{类型：钩子|技能|注释、描述、目标文件、任何来自第 2 阶段的详细信息，例如实际的测试/格式化命令}。阶段 4-7 消耗此队列。

## 第 4 阶段：编写 CLAUDE.md（如果用户选择项目或两者）

在项目根目录编写一个最小的 CLAUDE.md。每行都必须通过这个测试：“删除这个会导致克劳德犯错误吗？”如果没有，就剪掉。

**使用目标为 CLAUDE.md 的第 3 阶段首选项队列中的 `note` 条目**（团队级别注释） - 将每个条目作为简洁的行添加到最相关的部分中。这些是用户希望 Claude 遵循但不需要保证的行为（例如，“在实施之前提出计划”、“解释重构时的权衡”）。为第 5 阶段留下个人目标笔记。

包括：
- 克劳德无法猜测的构建/测试/lint 命令（非标准脚本、标志或序列）
- 与语言默认值不同的代码样式规则（例如，“优先选择类型而不是接口”）
- 测试说明和怪癖（例如，“使用以下命令运行单个测试：pytest -k 'test_name'”）
- 回购礼仪（分支命名、PR 约定、提交风格）
- 所需的环境变量或设置步骤
- 不明显的问题或架构决策
- 现有 AI 编码工具配置的重要部分（如果存在）（AGENTS.md、.cursor/rules、.cursorrules、.github/copilot-instructions.md、.windsurfrules、.clinerules）

排除：
- 逐个文件的结构或组件列表（克劳德可以通过阅读代码库发现这些）
- 克劳德已经知道的标准语言约定
- 通用建议（“编写干净的代码”，“处理错误”）
- 详细的 API 文档或长参考 — 使用 `@path/to/import` 语法（例如 `@docs/api-reference.md`）来按需内联内容，而不会膨胀 CLAUDE.md
- 经常更改的信息 — 使用 `@path/to/import` 引用源，以便 Claude 始终读取当前版本
- 长教程或演练（移至单独的文件并使用 `@path/to/import` 进行参考，或放入技能）
- 清单文件中显而易见的命令（例如，标准“npm test”、“cargo test”、“pytest”）

具体来说：“在 TypeScript 中使用 2 个空格缩进”比“正确设置代码格式”更好。

不要重复自己，也不要编造诸如“常见开发任务”或“开发提示”之类的部分 - 仅包含在您阅读的文件中明确找到的信息。文件前缀为：```
# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.
```
如果 CLAUDE.md 已经存在：阅读它，提出具体的更改作为差异，并解释为什么每个更改都会改进它。不要默默地覆盖。

对于具有多个问题的项目，建议将说明组织到 `.claude/rules/` 作为单独的重点文件（例如，`code-style.md`、`testing.md`、`security.md`）。它们会与 CLAUDE.md 一起自动加载，并且可以使用 `paths` frontmatter 将范围限定到特定文件路径。

对于具有不同子目录的项目（monorepos、多模块项目等）：提到可以添加子目录 CLAUDE.md 文件以用于特定于模块的指令（当 Claude 在这些目录中工作时，它们会自动加载）。如果用户愿意，可以主动创建它们。

## 第 5 阶段：写入 CLAUDE.local.md（如果用户选择个人或两者）

在项目根目录写入一个最小的 CLAUDE.local.md。该文件会与 CLAUDE.md 一起自动加载。创建后，将 `CLAUDE.local.md` 添加到项目的 .gitignore 中，以使其保持私有状态。

**使用目标为 CLAUDE.local.md 的第 3 阶段首选项队列中的 `note` 条目**（个人级别注释） - 将每个条目添加为简洁的行。如果用户在第一阶段选择仅限个人，则这是注释条目的唯一消费者。

包括：
- 用户的角色和对代码库的熟悉程度（以便 Claude 可以校准解释）
- 个人沙箱 URL、测试帐户或本地设置详细信息
- 个人工作流程或沟通偏好

保持简短——只包含能让 Claude 的响应对该用户明显更好的内容。

如果第 2 阶段发现多个 git 工作树并且用户确认他们使用同级/外部工作树（未嵌套在主存储库内）：向上文件遍历将不会从所有工作树中找到单个 CLAUDE.local.md。将实际的个人内容写入 `~/.claude/<project-name>-instructions.md` 并使 CLAUDE.local.md 成为导入它的单行存根：`@~/.claude/<project-name>-instructions.md`。用户可以将此单行存根复制到每个同级工作树。切勿将此导入放入项目 CLAUDE.md 中。如果工作树嵌套在主存储库中（例如，`.claude/worktrees/`），则不需要特殊处理 - 主存储库的 CLAUDE.local.md 会自动找到。

如果 CLAUDE.local.md 已经存在：阅读它，提出具体的添加，并且不要默默地覆盖。

## 第 6 阶段：建议并创建技能（如果用户选择“技能 + 挂钩”或“仅技能”）

技能增加了克劳德可以按需使用的能力，而不会在每次会话中都显得臃肿。

**首先，使用第 3 阶段首选项队列中的 `skill` 条目。** 每个排队的技能首选项都会成为根据用户描述定制的 SKILL.md。对于每个：
- 根据首选项命名（例如“verify-deep”、“session-report”、“deploy-sandbox”）
- 使用用户在采访中自己的话以及第二阶段发现的任何内容（测试命令、报告格式、部署目标）来编写正文。如果偏好映射到现有的捆绑技能（例如，`/verify`），请编写一项项目技能，在顶部添加用户的特定约束 - 告诉用户捆绑的技能仍然存在，并且他们的技能是附加的。
- 如果首选项未指定，请快速跟进（例如，“应验证深度运行哪个测试命令？”）**然后，当您发现以下情况时，建议队列之外的其他技能**：
- 特定任务的参考知识（子系统的约定、模式、风格指南）
- 用户希望直接触发的可重复工作流程（部署、修复问题、发布流程、验证更改）

对于每项建议的技能，请提供：名称、一行目的以及为什么它适合此存储库。

如果 `.claude/skills/` 已经存在技能，请先查看它们。不要覆盖现有技能——只提出新的技能来补充已有的技能。

在`.claude/skills/<skill-name>/SKILL.md`创建每个技能：```yaml
---
name: <skill-name>
description: <what the skill does and when to use it>
---

<Instructions for Claude>
```
默认情况下，用户（`/<skill-name>`）和克劳德都可以调用技能。对于具有副作用的工作流程（例如，`/deploy`、`/fix-issue 123`），请添加 `disable-model-invocation: true`，以便只有用户可以触发它，并使用 `$ARGUMENTS` 接受输入。

## 第 7 阶段：提出额外优化建议

告诉用户，既然 CLAUDE.md 和技能（如果选择）已经到位，您将建议一些额外的优化。

检查环境并询问您发现的每个差距（使用 AskUserQuestion）：

- **GitHub CLI**：运行 `which gh`（或 Windows 上的 `where gh`）。如果它丢失并且项目使用 GitHub（检查 github.com 的 `git remote -v`），请询问用户是否要安装它。说明 GitHub CLI 可让 Claude 直接帮助提交、拉取请求、问题和代码审查。

- **Linting**：如果第 2 阶段没有发现 lint 配置（项目语言没有 .eslintrc、ruff.toml、.golangci.yml 等），请询问用户是否希望 Claude 为此代码库设置 lint。解释 linting 可以及早发现问题，并为 Claude 提供对其自己编辑的快速反馈。

- **提案来源的挂钩**（如果用户选择“技能 + 挂钩”或“仅挂钩”）：使用第 3 阶段首选项队列中的 `hook` 条目。如果第 2 阶段找到格式化程序并且队列没有格式化挂钩，请提供编辑时格式化作为后备。如果用户在第一阶段选择“都不”或“仅技能”，则完全跳过此项目符号。

  对于每个钩子首选项（来自队列或格式化程序回退）：

  1、目标文件：默认基于Phase 1 CLAUDE.md选择——项目→`.claude/settings.json`（团队共享，已提交）；个人 → `.claude/settings.local.json`。仅询问用户是否在第一阶段选择了“两者”或者偏好不明确。对所有钩子询问一次，而不是针对每个钩子。

  2. 从首选项中选择事件和匹配器：
     -“每次编辑后”→ `PostToolUse` 与匹配器 `Write|Edit`
     -“当克劳德完成时”/“在我回顾之前”→ `Stop` 事件（在每个回合结束时触发 - 包括只读回合）
     - “运行 bash 之前”→ `PreToolUse` 与匹配器 `Bash`
     - “提交之前”（字面量 git-commit 门）→ **不是 hooks.json 钩子。** 匹配器无法通过命令内容过滤 Bash，因此无法仅针对 `git commit`。将其路由到 git 预提交挂钩（`.git/hooks/pre-commit`、husky、预提交框架）——提议编写一个。如果用户实际上的意思是“在我审查并提交 Claude 的输出之前”，那就是 `Stop` — 探测以消除歧义。
     探究偏好是否不明确。

  3. **加载钩子参考**（每次 `/init` 运行一次，在第一个钩子之前）：使用 `skill: 'update-config'` 和以 `[hooks-only]` 开头的参数调用技能工具，后跟您正在构建的内容的单行摘要 - 例如，`[hooks-only] Constructing a PostToolUse/Write|Edit format hook for .claude/settings.json using ruff`。这会将钩子架构和验证流程加载到上下文中。后续的钩子会重用它——不要重新调用。

  4. 按照技能的 **“构建 Hook”** 流程：重复数据删除检查 → 构建此项目 → 管道测试原始 → 包装 → 写入 JSON → `jq -e` 验证 → 实时验证（对于可触发匹配器上的 `Pre|PostToolUse`）→ 清理 → 移交。目标文件和事件/匹配器来自上面的步骤 1-2。在继续之前，先对每个“是”采取行动。

## 第 8 阶段：总结和后续步骤

回顾一下设置的内容——编写了哪些文件以及每个文件中包含的要点。提醒用户这些文件是一个起点：他们应该检查和调整它们，并且可以随时再次运行 `/init` 来重新扫描。

然后告诉用户，您将根据您的发现引入更多建议，以优化他们的代码库和 Claude 代码设置。将它们呈现为单个、格式良好的待办事项列表，其中每个项目都与此存储库相关。将最有影响力的项目放在第一位。

构建列表时，请完成这些检查并仅包含适用的内容：
- 如果检测到前端代码（React、Vue、Svelte 等）：`/plugin install frontend-design@claude-plugins-official` 提供 Claude 设计原则和组件模式，以便生成精美的 UI； `/plugin install playwright@claude-plugins-official` 让克劳德启动一个真正的浏览器，截图它构建的内容，并修复视觉错误本身。
- 如果您在第 7 阶段发现了差距（缺少 GitHub CLI、缺少 linting）并且用户拒绝：将它们列在此处，并附上一行理由解释为什么每项都有帮助。
- 如果测试缺失或稀疏：建议设置一个测试框架，以便 Claude 可以验证自己的更改。
- 为了帮助您使用评估创建技能并优化现有技能，Claude Code 有一个您可以安装的官方技能创建器插件。使用 `/plugin install skill-creator@claude-plugins-official` 安装它，然后运行 ​​`/skill-creator <skill-name>` 来创建新技能或完善任何现有技能。 （始终包括这一点。）
- 使用 `/plugin` 浏览官方插件 - 这些捆绑技能、代理、挂钩和 MCP 服务器可能对您有用。您还可以创建自己的自定义插件以与其他人共享。 （始终包括这一点。）
