# /stuck — 诊断冻结/缓慢的 Claude 代码会话

用户认为该计算机上的另一个 Claude Code 会话被冻结、卡住或非常慢。调查并向 #claude-code-feedback 发布报告。

## 寻找什么

扫描其他 Claude Code 进程（不包括当前进程 - PID 位于 `process.pid` 中，但对于 shell 命令，只需排除您在运行此提示符时看到的 PID）。进程名称通常为 `claude`（已安装）或 `cli`（本机开发版本）。

会话卡住的迹象：
- **持续高 CPU (≥90%)** — 可能是无限循环。采样两次，间隔 1-2 秒，以确认它不是瞬态尖峰。
- **进程状态 `D`（不间断睡眠）** — 通常是 I/O 挂起。 `ps` 输出中的 `state` 列；第一个字符很重要（忽略 `+`、`s`、`<` 等修饰符）。
- **进程状态 `T`（已停止）** — 用户可能意外按下 Ctrl+Z。
- **进程状态 `Z`（僵尸）** — 父进程没有收获。
- **非常高的 RSS (≥4GB)** — 可能存在内存泄漏，导致会话缓慢。
- **卡住子进程** — 挂起的 `git`、`node` 或 shell 子进程可以冻结父进程。检查每个会话的 `pgrep -lP <pid>`。

## 调查步骤

1. **列出所有 Claude Code 进程** (macOS/Linux)：   ```
   ps -axo pid=,pcpu=,rss=,etime=,state=,comm=,command= | grep -E '(claude|cli)' | grep -v grep
   ```
筛选到 `comm` 为 `claude` 或（`cli` 并且命令路径包含“claude”）的行。

2. **对于任何可疑的事情**，收集更多背景信息：
   - 子进程：`pgrep -lP <pid>`
   - 如果 CPU 高：1-2 秒后再次采样以确认其持续
   - 如果子进程看起来挂起（例如 git 命令），请用 `ps -p <child_pid> -o command=` 记下其完整命令行
   - 检查会话的调试日志，如果您可以推断会话 ID：`~/.claude/debug/<session-id>.txt`（最后几百行通常显示它在挂起之前正在做什么）

3. **考虑堆栈转储**以实现真正冻结的进程（高级，可选）：
   - macOS：`sample <pid> 3` 提供 3 秒本机堆栈示例
   - 这很大——只有当进程明显挂起并且你想知道*为什么*时才抓住它

## 报告

**仅当您确实发现有问题时才发布到 Slack。** 如果每个会话看起来都很正常，请直接告诉用户 — 不要向频道发布解除警报。

如果您确实发现会话卡住/缓慢，请使用 Slack MCP 工具发布到 **#claude-code-feedback**（频道 ID：`C07VBSHV7EV`）。如果尚未加载，请使用 ToolSearch 查找 `slack_send_message`。

**使用两条消息结构**来保持频道可扫描：

1. **顶级消息** — 一小行：主机名、Claude 代码版本和简洁的症状（例如“会话 PID 12345 固定在 100% CPU 上 10 分钟”或“git 子进程挂起在 D 状态”）。没有代码块，没有细节。
2. **线程回复** — 完整的诊断转储。将顶级消息的 `ts` 传递为 `thread_ts`。包括：
   - PID、CPU%、RSS、状态、正常运行时间、命令行、子进程
   - 您对可能出现问题的诊断
   - 相关调试日志尾部或 `sample` 输出（如果您捕获了它）

如果 Slack MCP 不可用，请将报告格式化为消息，用户可以将其复制粘贴到 #claude-code-feedback 中（并让他们知道自己线程化详细信息）。

## 注释
- 不要终止或向任何进程发出信号 - 这仅用于诊断。
- 如果用户给出了一个论点（例如，特定的 PID 或症状），请首先关注那里。
