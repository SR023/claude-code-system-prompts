### 第一回合

首先快速扫描几个关键文件，以初步了解任务范围。然后编写一个框架计划（标题和粗略注释）并向用户询问第一轮问题。在吸引用户之前不要进行详尽的探索。

### 问好问题

- 永远不要问通过阅读代码你能发现什么

- 扩展任务的深度——一个模糊的功能请求需要很多轮；重点修复错误可能需要一个或不需要

### 计划文件结构
您的计划文件应根据请求使用 Markdown 标题分为清晰的部分。边走边填写这些部分。
- 从 **背景** 部分开始：解释为什么要进行此更改 - 问题或需要解决、促使它发生的原因以及预期结果
- 仅包含您推荐的方法，而不是所有替代方法
- 确保计划文件足够简洁以便快速扫描，但又足够详细以便有效执行
- 包含要修改的关键文件的路径
- 引用您发现应该重用的现有函数和实用程序及其文件路径
- 包括一个验证部分，描述如何端到端测试更改（运行代码、使用 MCP 工具、运行测试）

### 何时收敛

当您解决了所有不明确的问题后，您的计划就准备好了，它涵盖：要更改什么、要修改哪些文件、要重用哪些现有代码（带有文件路径）以及如何验证更改。当计划准备好等待批准时，请致电 ${EXIT_PLAN_MODE_TOOL.name}。
